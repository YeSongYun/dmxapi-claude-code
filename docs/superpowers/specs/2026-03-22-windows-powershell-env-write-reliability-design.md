# Windows PowerShell 环境变量写入可靠性修复设计文档

**日期**: 2026-03-22
**状态**: 已审批

---

## 背景

用户在 Windows PowerShell 中通过以下方式运行安装脚本时：

```powershell
iwr -useb https://cnb.cool/dmxapi/dmxapi_claude_code/-/git/raw/main/install.ps1 | iex
```

遇到两个连带问题：

1. 配置流程失败后表现为“闪退”或快速结束，错误原因不清晰
2. 相关环境变量未成功写入当前用户的 Windows User 级环境变量中，系统环境变量列表里为空

根据 README 当前约定，Windows 平台的配置结果应写入 **User 级环境变量**，用户重新打开终端后即可生效。因此这次修复需要同时解决：

- User 级持久化写入不可靠
- 写入失败时缺少明确错误暴露
- PowerShell 安装入口无法清楚反映 exe 的失败状态

---

## 目标

本次修复目标如下：

1. Windows 端环境变量只写入 **User 级**
2. 每个环境变量写入后立即执行回读校验
3. 任一关键变量写入失败或校验失败时，整个配置流程明确失败
4. PowerShell 安装脚本在失败时显示清晰错误，不再以“秒退出”形式吞掉信息
5. 让 PowerShell、CMD、直接运行 exe 三种入口最终共享同一套 Go 写入逻辑

---

## 修复范围

### 包含内容

#### 1. Go 主程序中的 Windows 环境变量写入逻辑

- 固定写入当前用户（User）环境变量
- 对设置与删除动作都增加写后回读验证
- 明确区分：
  - 写入动作失败
  - 写入后校验失败
- 任一关键变量失败时，整体返回错误，禁止“部分失败但表面成功”

#### 2. PowerShell 安装脚本 `install.ps1`

- 调整当前的退出行为，避免在 `iwr | iex` 场景下直接结束宿主会话而导致错误不可见
- 在 exe 结束后检查退出码
- 如果 exe 非 0 退出，显示明确失败提示和退出码
- 成功时正常清理临时文件并结束

#### 3. 测试与文档

- 为 Windows 环境变量写入链路补充测试
- 覆盖写入成功、写入失败、回读不一致、删除失败等关键场景
- 必要时更新 README 中 Windows 配置行为说明，使其与实际实现一致

### 不包含内容

- 不增加 Machine 级环境变量写入
- 不增加管理员权限逻辑
- 不改变 Linux / macOS 的环境变量写入流程
- 不引入新的安装入口或新的配置模式

## 设计方案

### 方案选择

考虑过三种方式：

#### 方案 A：仅修复 `install.ps1` 的错误展示

优点：改动小，可以解决“闪退看不到错误”。

缺点：不能根治 Go 主程序里真实的写入失败问题，只能把失败暴露出来。

#### 方案 B：仅增强 Go 主程序的 Windows 写入逻辑

优点：能根治 User 级环境变量写入可靠性问题。

缺点：PowerShell 入口的失败体验仍然可能很差，用户依旧可能只感知为“闪退”。

#### 方案 C：同时修复 Go 写入逻辑与 `install.ps1`（采用）

优点：

- 同时解决“写不进去”和“失败不可见”两个问题
- PowerShell / CMD / 直接 exe 三种入口行为更一致
- 修复面集中在现有边界内，不引入额外复杂度

因此采用 **方案 C**。

## 核心实现

### 一、Windows User 级环境变量写入闭环

Windows 环境变量写入逻辑需要从“执行一次写命令”升级为“写入 + 校验”的闭环流程。

对每个需要持久化的环境变量执行以下步骤：

1. 判断当前动作是“设置”还是“删除”
2. 执行对应的 User 级环境变量持久化写入
3. 立即从 User 级来源回读该变量
4. 校验回读值是否与期望一致
5. 任一步失败立即返回错误

这样可以避免以下问题：

- 写入命令返回成功，但实际没有写入注册表
- 删除动作执行后，变量实际上仍然存在
- 只有部分变量写入成功，但最终流程仍然显示成功

### 二、统一 User 级写入目标

本次修复明确约束：

- 仅写入当前用户（User）环境变量
- 不尝试 Machine 级
- 不要求管理员权限

这样能保证：

- 与 README 当前承诺一致
- 在普通用户权限下行为稳定
- 避免因权限不足引入新的分支逻辑

### 三、可测试抽象

为了使测试稳定，Windows 的持久化环境变量操作不应直接散落在配置流程中，而应收口为最小操作边界，例如：

- `setUserEnv(name, value)`
- `removeUserEnv(name)`
- `getUserEnv(name)`

主流程只负责：

- 计算要写哪些变量
- 调用上述能力进行设置或删除
- 对结果做一致性校验

测试时可替换为 fake / stub，从而稳定覆盖失败分支，而不依赖真实 Windows 注册表环境。

## 错误处理设计

### Go 主程序错误输出

当某个变量在 Windows 上写入失败时，错误信息至少应包含：

- 变量名
- 当前动作（设置或删除）
- 失败阶段（写入失败或校验失败）
- 原始错误原因或实际回读值

示例风格：

```text
failed to set user env ANTHROPIC_BASE_URL: <reason>
failed to verify user env ANTHROPIC_AUTH_TOKEN: expected <expected>, got <actual>
failed to remove user env ANTHROPIC_MODEL: <reason>
```

不要求文案完全一致，但要求诊断粒度达到该级别。

### PowerShell 安装脚本错误输出

`install.ps1` 当前在结尾直接 `exit`，这在 `iwr | iex` 场景下容易导致会话快速结束，用户看不到失败原因。

修复后应满足：

1. 启动 exe 并等待结束
2. 获取 exe 的退出码
3. 若退出码非 0：
   - 明确显示配置失败
   - 显示退出码
   - 保证错误信息在终端里可见，而不是被静默吞掉
4. 若退出码为 0：
   - 清理临时文件
   - 正常结束脚本

这里的 PowerShell 脚本只负责：

- 下载
- 启动
- 呈现失败状态

真正的配置成功与否仍由 Go 主程序负责判定。

## 验证标准

### 成功路径

在 Windows PowerShell 中执行安装脚本并完成配置后，应满足：

1. 程序返回成功
2. User 级环境变量中存在预期值
3. 新开 PowerShell 后，`$env:ANTHROPIC_BASE_URL` 等变量可直接读取到

### 失败路径

若某个关键变量无法写入或写后回读不一致，应满足：

1. 程序整体返回失败
2. 用户能看到失败变量与失败原因
3. 不再出现“流程结束了但环境变量实际为空”的假成功

### 一致性

无论入口是：

- PowerShell 安装脚本
- CMD 安装脚本
- 直接运行 exe

最终都应依赖同一套 Go 环境变量写入与校验逻辑，不因入口不同而出现行为分叉。

## 测试计划

### 单元测试

补充与 Windows 写入逻辑相关的测试，至少覆盖以下场景：

1. **设置成功并校验通过**
   - 写入成功
   - 回读值与期望一致
   - 整体成功

2. **设置动作失败**
   - `setUserEnv` 返回错误
   - 主流程立即失败

3. **设置后回读不一致**
   - 写入动作成功
   - `getUserEnv` 返回空值、旧值或其他错误值
   - 主流程判定为校验失败

4. **删除成功后回读仍存在**
   - 删除动作成功
   - 但回读结果仍存在旧值
   - 主流程失败

5. **多个变量中某一个失败**
   - 前几个变量成功
   - 某一个变量失败
   - 整体失败，不允许假成功

### 手工验证

建议在 Windows 上执行以下验证步骤：

1. 先清理已有相关 User 环境变量
2. 运行 `install.ps1`
3. 输入配置项完成配置
4. 立即验证 User 级写入：

```powershell
[System.Environment]::GetEnvironmentVariable("ANTHROPIC_BASE_URL", "User")
```

5. 关闭并重新打开 PowerShell，再验证：

```powershell
echo $env:ANTHROPIC_BASE_URL
```

6. 人工制造失败场景时，确认程序能显示清晰错误，而非快速退出

## 成功定义

本次修复完成的标志是：

- Windows PowerShell 下配置环境变量可以稳定写入到 User 级
- 写入失败时用户可以看到明确错误，而不是“闪退”
- 三种入口共享同一套可靠的 Windows 持久化写入逻辑
- 测试能够覆盖主要失败分支，后续不易回归
